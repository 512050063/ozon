# 商品库上架确认弹窗设计

**目标**

为商品库模块新增“上架确认弹窗”，在真正调用 Ozon 上架接口前，要求用户选择目标店铺、选择定价策略、确认最终上架价格，并完成一次前后端双重的商品信息规范检查。

**范围**

- 仅覆盖 `商品库` 模块上架流程
- 单次上架只允许选择 `1` 个 Ozon 店铺
- 定价策略只影响 `本次上架提交`
- 商品库原始价格不因定价策略自动回写

**非目标**

- 不实现多店铺同时上架
- 不改造 `商品管理` 模块
- 不新增复杂的定价策略编辑能力
- 不处理 Ozon 上架后的价格同步问题

## 现状

- 商品库列表当前点击“上架”后，前端直接调用 `/product-supply/:id/list-to-ozon`
- 当前上架接口不接收店铺、定价策略、最终价格参数
- 项目已有：
  - Ozon 店铺配置与查询能力
  - 定价策略 CRUD 能力
  - 商品库本地数据与模板数据
  - Ozon 上架提交与异步状态轮询

当前缺失的是“上架前的决策和校验层”。

## 用户流程

1. 用户在商品库列表点击“上架”
2. 前端不直接提交，而是打开“上架确认弹窗”
3. 弹窗加载：
   - 当前商品基础信息
   - 可用店铺列表
   - 用户定价策略列表
   - 初步检查结果
   - 建议上架价格
4. 用户选择店铺、选择定价策略
5. 系统展示：
   - 商品库价格
   - 策略计算建议价
   - 最终上架价格输入框
6. 用户确认最终价格
7. 系统展示检查结果：
   - 通过项
   - 警告项
   - 阻止提交项
8. 若存在阻止项，则禁止提交
9. 若只有警告项，则允许提交，但用户明确确认后继续
10. 后端再次校验通过后，才真正调用 Ozon 上架接口

## 弹窗结构

### 1. 上架目标

- 店铺选择：单选下拉
- 定价策略选择：单选下拉
- 若没有可用店铺：
  - 直接显示“请先配置并启用 Ozon 店铺”
- 若没有定价策略：
  - 直接显示“请先创建定价策略”

### 2. 价格预览

展示以下信息：

- 商品库当前价格
- 所选定价策略名称
- 建议上架价格
- 最终上架价格

交互规则：

- 默认最终上架价格 = 建议上架价格
- 用户可手动修改最终上架价格
- 最终价格必须大于 0
- 价格按字符串提交给 Ozon，避免重复出现数值类型错误

### 3. 提交前检查

检查结果分组展示：

- `基础信息`
- `模板信息`
- `媒体与图片`
- `上架参数`

每条检查项包含：

- 名称
- 状态：`pass | warning | error`
- 文案说明

若存在 `error`，提交按钮禁用。

## 校验规则

### 阻止提交（error）

- 未选择店铺
- 未选择定价策略
- 最终上架价格为空或 `<= 0`
- 商品名称为空
- 主图为空
- 商品类目为空 / `descriptionCategoryId` 缺失
- 品牌为空
- 型号为空
- 包装长宽高任一为空或 `<= 0`
- 重量为空或 `<= 0`
- 必填变体属性未填写
- 必填隐藏属性未填写
- 编辑单 SKU 时 `offer_id` 缺失
- 新增多 SKU 时任一 SKU 的 `offer_id` 缺失或重复
- 图片无法转换成公网可访问 URL

### 允许提交但提示（warning）

- 描述为空
- 商品名称过短
- 最终上架价格与商品库原价差异过大
- 当前商品存在可选属性未填
- 商品没有条形码

### 通过（pass）

- 对应检查项符合要求

## 定价策略计算

弹窗需要基于选中的定价策略，生成一个“建议上架价格”。

建议价格计算器需要复用 `pricingStrategy` 已有字段：

- `basePrice`
- `shippingPrice`
- `tariffRate`
- `profitRate`
- `platformFeeRate`
- `otherCost`

本次设计要求：

- 在后端集中实现计算逻辑
- 前端只展示计算结果和计算明细
- 同一策略对同一商品的建议价应稳定可复现

最终上架价格允许人工覆盖建议价。

## 接口设计

### 1. 上架预览接口

`POST /product-supply/:id/listing-preview`

**请求**

```json
{
  "storeId": 2,
  "pricingStrategyId": 3,
  "finalPrice": 1299
}
```

**响应**

```json
{
  "success": true,
  "data": {
    "product": {},
    "stores": [],
    "pricingStrategies": [],
    "pricing": {
      "productPrice": 99,
      "suggestedPrice": 1299,
      "finalPrice": 1299,
      "currencyCode": "RUB",
      "strategyName": "默认策略",
      "breakdown": []
    },
    "checks": [
      {
        "key": "mainImage",
        "group": "媒体与图片",
        "label": "主图",
        "status": "pass",
        "message": "主图已填写"
      }
    ],
    "canSubmit": true,
    "blockingCount": 0,
    "warningCount": 1
  }
}
```

说明：

- `storeId / pricingStrategyId / finalPrice` 在首次打开时可以为空
- 若为空，接口仍返回店铺和策略列表，以及未完成选择的检查结果

### 2. 上架提交接口

保留原路径，但扩展参数：

`POST /product-supply/:id/list-to-ozon`

**请求**

```json
{
  "storeId": 2,
  "pricingStrategyId": 3,
  "finalPrice": 1299
}
```

后端行为：

1. 校验商品归属
2. 校验商品状态允许上架
3. 校验所选店铺属于当前用户且可用
4. 校验所选定价策略属于当前用户
5. 重跑一次 `listing-preview` 同源校验
6. 若存在阻止项，拒绝提交
7. 用 `finalPrice` 生成 Ozon payload
8. 提交 Ozon import
9. 更新本地商品库状态为 `listing`
10. 记录本次上架使用的店铺、策略、最终价格和检查结果摘要

## 数据落库

本次至少需要补充记录本次上架提交的上下文，避免后续无法追踪：

- 上架店铺 ID
- 定价策略 ID
- 本次最终上架价格
- 上架前校验摘要

若现有 `products_supply` 表没有合适字段，则新增轻量字段即可，不引入新表。

## 前端实现边界

新增一个独立弹窗组件，挂在商品库页面：

- 列表点击“上架”时打开
- 不污染添加/编辑商品抽屉逻辑
- 弹窗内部自己负责：
  - 数据加载
  - 选择店铺
  - 选择策略
  - 价格预览
  - 展示检查项
  - 确认提交

## 后端实现边界

后端新增一组商品库上架预检查能力：

- 统一计算建议价
- 统一输出检查项
- 上架预览和真正提交共用同一套校验逻辑

这样可以避免：

- 前后端校验标准不一致
- 前端看起来能上架，后端又拒绝
- 上架逻辑散在多个控制器里

## 错误处理

### 前端

- 店铺/策略加载失败：弹窗内展示错误态，禁止提交
- 预检查失败：弹窗内展示错误信息
- 上架失败：列表刷新并展示后端错误文案

### 后端

- 店铺不存在 / 不属于当前用户：返回 400
- 定价策略不存在 / 不属于当前用户：返回 400
- 商品信息不完整：返回 400，并附检查项
- Ozon 提交失败：保留失败原因并截断写库

## 测试要求

### 后端

- 预检查逻辑：
  - 缺图
  - 缺品牌
  - 缺类目
  - 缺必填属性
  - 价格无效
  - 多 SKU 重复 `offer_id`
- 模板过滤与上架属性生成回归继续保留
- 店铺 / 策略权限校验

### 前端

- 弹窗打开关闭
- 店铺和策略选择后刷新预览
- 最终价格手动修改
- 有阻止项时按钮禁用
- 下拉长选项显示不溢出

## 兼容性

- 保持现有轮询上架状态逻辑不变
- 保持商品库列表结构不变
- 保持原有 `/product-supply/:id/list-to-ozon` 路径不变，仅扩展请求体

## 风险与约束

- 当前仓库存在与本任务无关的后端 TypeScript 类型错误，实施时需要避免误判为本次改动引入
- 定价公式若历史上没有统一定义，本次需要固定成一套后端实现
- 旧商品库数据可能存在字段不完整，需要靠预检查拦截
